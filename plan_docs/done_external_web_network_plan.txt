# F7LY 外部网页访问网络实现计划

> 给 agent/工程师执行时使用：建议按任务顺序逐项推进。每个任务完成后先跑本任务验收命令，再继续下一项。不要未经用户允许提交 commit。

**目标：** 在当前已经跑通 127.0.0.1 TCP/UDP loopback 的基础上，做出一条能从用户态访问外部 HTTP 网页并读取数据的完整网络链路。

**架构：** 保留当前 socket_file 的 loopback 快路径，用于本机 127.0.0.1 和现有 LTP/iperf/netperf loopback 测试；把非 loopback IPv4 客户端连接接到 ONPS 协议栈；ONPS 再通过 virtio-net 驱动收发真实以太网帧。第一目标是 RISC-V + QEMU user networking 下的 HTTP GET，随后扩展 DNS、BusyBox wget、LoongArch。

**技术栈：** F7LY C++23 freestanding 内核、Linux socket ABI、fs::socket_file、kernel/net/onpstack、virtio-net、QEMU user net、用户态 initcode 回归测试。

---

## 0. 当前事实和边界

### 已经有的能力

- RISC-V/musl 下 127.0.0.1 TCP payload 已经能真实传输。
- RISC-V/musl 下 127.0.0.1 UDP payload 已经能真实传输。
- LTP socket 小集合已经能通过：socket01/socket02/accept01/accept03/accept4_01/bind01/listen01/connect01/sendto01/recvfrom01/recvmsg01/socketpair02。

### 当前不能访问外网的原因

- `kernel/proc/proc_manager.cc` 中 `net::init_network_stack()` 仍是注释状态，ONPS/virtio-net 默认不会启动。
- `kernel/fs/vfs/file/socket_file.cc` 目前把 AF_INET 基本限制在 loopback/0.0.0.0，非 127.0.0.1 的 connect 会返回 `-ENETUNREACH`。
- `kernel/net/drivers/virtio_net_adapter.cc` 的 `virtio_recv_thread()` 目前打印“未实现”，没有把 virtio-net 收到的 Ethernet frame 交给 ONPS。
- `kernel/net/drivers/virtio_net.cc` 的 TX/RX descriptor 回收、中断/轮询、MAC 读取、MMIO 地址都需要重新验证。
- `kernel/net/drivers/virtio_net_adapter.cc` 现在写死 `192.168.1.2/24`、网关 `192.168.1.1`、DNS `8.8.8.8`，这和 QEMU `-netdev user` 默认网络不匹配。QEMU user net 最小静态配置应优先使用：
  - guest IP：`10.0.2.15`
  - netmask：`255.255.255.0`
  - gateway：`10.0.2.2`
  - DNS：`10.0.2.3`
- HTTPS 不属于第一阶段目标。先支持 HTTP 明文网页数据；HTTPS 需要 TLS、证书、时间等用户态能力，放到后续。

### 最小成功标准

第一阶段成功不要定义成“wget 全部可用”，而是定义成：

- 用户态 `socket(AF_INET, SOCK_STREAM, 0)` 成功。
- 用户态 `connect()` 到非 loopback IPv4:80 成功。
- 用户态 `send()` 发出 HTTP GET。
- 用户态 `recv()` 收到以 `HTTP/` 开头的数据，且长度大于 0。
- QEMU 日志里能看到 virtio-net TX/RX 计数增长。

### 测例检查点使用原则

外网实现过程中，测例要分层使用：

- 先用项目自写 smoke 证明单个能力点，例如 loopback payload、HTTP gateway、HTTP external IP。
- 再用 LTP socket syscall 类测例证明 Linux ABI 没写歪。
- 再用 iperf/netperf 证明 loopback 并发和吞吐稳定。
- 最后再用 LTP network stress、DNS stress、HTTP stress 这类环境依赖很重的测例。

不要把 `accept01`、`bind01` 这类 errno 型 PASS 当成外网可用；也不要把 `dns-stress.sh`、`http-stress.sh` 这类后期压力脚本放到早期门禁里。

当前建议保留为每次网络改动后的基础门禁：

- `network_loopback_smoke()`
- `network_fork_loopback_smoke()`
- LTP socket 小集合：socket01/socket02/accept01/accept03/accept4_01/bind01/listen01/connect01/sendto01/recvfrom01/recvmsg01/socketpair02

当前已有 RISC-V/musl 证据：

- `logs/output_r_20260526-134750_network-loopback-smoke_QEMU_MEM-1G_timeout-5m.txt`
- `logs/output_r_20260526-134912_network-ltp-socket-subset_QEMU_MEM-1G_timeout-5m.txt`
- `logs/output_r_20260526-135203_network-fork-loopback-smoke_QEMU_MEM-1G_timeout-5m.txt`

## 1. 文件职责规划

### 需要修改的内核文件

- `kernel/proc/proc_manager.cc`
  - 在第一个普通进程上下文中启动网络栈。
  - 当前合适位置是 `ProcessManager::fork_ret()` 中 `filesystem_init()` 后面，那里已经有注释掉的 `net::init_network_stack()`。

- `kernel/net/f7ly_network.hh`
- `kernel/net/f7ly_network.cc`
  - 作为网络栈总入口，负责初始化 ONPS、virtio-net adapter、状态查询、计数器输出。
  - 增加“初始化成功/失败原因”和“是否已经可用于外部 socket”的查询接口。

- `kernel/net/network_config.hh`
- `kernel/net/network_config.cc`
  - 新增文件。
  - 保存默认 IPv4 配置，先支持 QEMU user net 静态配置。
  - 后续如要支持 DHCP，只替换这里和 adapter 配置来源。

- `kernel/net/drivers/virtio_net.hh`
- `kernel/net/drivers/virtio_net.cc`
  - 修复 RISC-V virtio-net 初始化、TX、RX、descriptor 回收。
  - 优先实现 polling fallback，再接中断。
  - 增加最小计数器：tx_submitted、tx_completed、rx_received、rx_dropped、rx_recycled、send_errors。

- `kernel/net/drivers/virtio_net_adapter.hh`
- `kernel/net/drivers/virtio_net_adapter.cc`
  - 把 virtio-net 驱动接到 ONPS Ethernet 层。
  - 实现 RX pump：从 virtio-net 取帧，调用 `ethernet_ii_recv(netif, packet, len)`。
  - 使用 `network_config` 的 QEMU user net 配置，不再写死 `192.168.1.x`。

- `kernel/devs/dtb.hh`
- `kernel/devs/dtb.cc`
  - 增加查找 virtio-mmio net 设备的辅助函数。
  - 目标是避免长期依赖 `VIRTIO_NET_MMIO_BASE 0x10008000` 这种猜测值。

- `kernel/mem/memlayout.hh`
- `kernel/mem/virtual_memory_manager.cc`
  - 如果最终确认 virtio-net MMIO 地址固定或来自 DTB，需要保证对应 MMIO 页被映射。
  - 短期可继续映射 `VIRTIO1`/探测出的 net MMIO；长期以 DTB 结果为准。

- `kernel/trap/riscv/plic.cc`
- `kernel/trap/riscv/trap.cc`
  - 第二阶段接 virtio-net 中断时修改。
  - 第一阶段允许 polling 跑通 HTTP，不强制先接中断。

- `kernel/fs/vfs/file/socket_file.hh`
- `kernel/fs/vfs/file/socket_file.cc`
  - 增加 socket backend 概念：loopback backend 和 onps backend。
  - loopback 地址继续走当前内核内队列。
  - 非 loopback IPv4 客户端 TCP/UDP 走 ONPS。

- `kernel/sys/syscall_handler.cc`
  - 原则上少改 syscall 层；参数拷贝和 Linux errno 仍留在这里。
  - 不要在 syscall 层直接操作 ONPS，避免重新出现 syscall/socket_file 双路径分裂。

- `kernel/fs/vfs/virtual_fs.cc`
- `kernel/fs/vfs/file/virtual_file.hh`
- `kernel/fs/vfs/file/virtual_file.cc`
  - DNS 阶段增加 `/etc/resolv.conf` 和必要的 `/etc/hosts` 虚拟文件，方便 musl/glibc 的 `getaddrinfo()` 和 BusyBox `wget`。

### 需要修改的用户态文件

- `user/deps/user.hh`
  - 声明新增网络 smoke 测试函数。

- `user/syscall_lib/syscall.cc`
  - 如果已有 socket/send/recv/connect 封装够用，则不新增。
  - 如果缺少 `getaddrinfo` 无关的简单测试 helper，可只在 `user_test.cc` 内部写。

- `user/user_lib/user_test.cc`
  - 新增 `network_http_gateway_smoke()`：连宿主机 `10.0.2.2:18080`，用于确定 virtio/QEMU user net 的 guest->host 链路。
  - 新增 `network_http_external_ip_smoke()`：连一个外部 HTTP IP:80，验证真实出网。
  - 新增 `network_http_dns_smoke()`：通过域名访问 HTTP，验证 DNS/`getaddrinfo`/UDP。

- `user/app/initcode-rv.cc`
  - 调试时临时切入口跑单项测试。
  - 每次跑完必须恢复原入口，避免污染回归。

- `user/app/initcode-la.cc`
  - LoongArch 阶段使用。
  - 当前宿主环境还缺 `debugfs`，要先解决运行环境再跑 LoongArch。

### 可选修改

- `Makefile`
  - 增加 `QEMU_NETDEV_ARGS` 或 `QEMU_NET_MODE`，方便在 user net、hostfwd、tap 之间切换。
  - 第一阶段可以不改 Makefile，继续用现有 `-netdev user,id=net`。

## 2. 阶段 A：建立可重复的外网测试入口

目标：先写出会失败的测试，明确当前失败点。不要直接修代码后再补测试。

### Task A1：新增 HTTP by IP 测试函数

**文件：**

- 修改：`user/user_lib/user_test.cc`
- 修改：`user/deps/user.hh`

**实现思路：**

在 `user/user_lib/user_test.cc` 增加一个小工具函数，流程固定为：

1. `socket(AF_INET, SOCK_STREAM, 0)`
2. `connect(ip, port)`
3. `send("GET / HTTP/1.0\r\nHost: host.local\r\nConnection: close\r\n\r\n")`
4. 循环 `recv()`，只要读到任意 `HTTP/` 开头的数据就 PASS
5. 超过固定次数还没读到数据就 FAIL

建议函数名：

- `int network_http_gateway_smoke(void);`
- `int network_http_external_ip_smoke(void);`

第一版只做 IPv4 数字地址，不做 DNS。

`network_http_gateway_smoke()` 默认目标：

- IP：`10.0.2.2`
- Port：`18080`
- Host header：`host.local`

这个目标用于配合宿主机本地 HTTP server，验证 guest -> host 网络链路。它不等于真实外网，但非常稳定，适合作为第一道门。

`network_http_external_ip_smoke()` 默认目标：

- 目标：HTTP 明文网站的 IPv4 地址，端口 80
- Host header：该网站域名
- PASS 条件：收到 `HTTP/` 状态行或任意非空 HTTP 响应头

不要把某个公共网站 IP 写死成唯一真理。外部网站 IP 可能变化，测试函数里要把失败日志打印清楚：目标 IP、connect 返回值、send 返回值、recv 返回值。

**验收命令：**

临时把 `user/app/initcode-rv.cc` 改为：

```cpp
#include "user.hh"

extern "C"
{
    __attribute__((section(".text.startup"))) int main()
    {
        network_http_gateway_smoke();
        shutdown();
        return 0;
    }
}
```

然后执行：

```bash
make build ARCH=riscv
```

预期：

- 当前还没有外部网络实现时，build 应通过。
- 运行测试大概率会失败在 `connect()`，这正是后续任务要修的红灯。

运行日志模板：

```bash

mkdir -p logs
log="logs/output_r_${ts}_network-http-gateway-smoke_QEMU_MEM-1G_timeout-5m.txt"

  echo "run_at=${ts}"

  echo "scope=network-http-gateway-smoke"
  echo "cmd=timeout 5m make run r QEMU_MEM=1G"
  echo "git_branch=$(git branch --show-current 2>/dev/null || true)"
  echo "git_head=$(git rev-parse --short HEAD 2>/dev/null || true)"
  echo "---- output ----"

  echo "exit_code=$?"

echo "$log"
```

注意：如果要在宿主机起本地 HTTP server，按项目 Python 规则，需要先准备并激活 venv：

```bash

source .venv/bin/activate
which python
python -m http.server 18080 --bind 0.0.0.0
```

### Task B2：启用网络栈初始化

**文件：**

- 修改：`kernel/proc/proc_manager.cc`
- 修改：`kernel/net/f7ly_network.cc`

**步骤：**

1. 在 `kernel/proc/proc_manager.cc` 引入 `net/f7ly_network.hh`。
2. 在 `ProcessManager::fork_ret()` 第一次进入、`filesystem_init()` 和 stdio device 初始化之后调用：

4. 初始化失败不能 panic，除非是明确的内核内存破坏；因为外网失败不能影响普通回归启动。

**验收：**

临时跑一个只启动后 shutdown 的 initcode，日志中应该看到：

- ONPS 初始化开始

- 如果 virtio-net 设备找不到，要明确打印设备探测失败原因
- 系统仍能 `shutdown()`

ONPS 和 virtio-net 初始化接入后，确认 Linux socket ABI 没被破坏。

**建议纳入的 LTP 测例：**

## 4. 阶段 C：修通 RISC-V virtio-net 收发

目标：能从 guest 发出以太网帧，也能收到返回帧。先用 polling，后接中断。

### Task C1：确认 virtio-net MMIO 地址来源

**文件：**

- 修改：`kernel/devs/dtb.hh`
- 修改：`kernel/devs/dtb.cc`
- 修改：`kernel/net/drivers/virtio_net.cc`
- 修改：`kernel/net/drivers/virtio_net.hh`

**实现要求：**

优先从 DTB 查找 `virtio,mmio` 节点：

- 读取节点 `reg`
- 读取节点 `interrupts`
- 读取 MMIO magic/version/device_id
- 选择 `device_id == 1` 的节点作为 virtio-net
- `device_id == 2` 是 virtio-blk，不要误用

如果 DTB 辅助函数暂时没写完，短期允许 fallback 到 `VIRTIO1 0x10002000`，但必须打印：

```text
[virtio_net] DTB probe failed, fallback to VIRTIO1=0x10002000
```

不要长期使用当前 `0x10008000` 作为唯一地址，除非日志证明 QEMU DTB 里 net 设备确实在这里。

**验收：**

日志中必须能看到：

- virtio-net 使用的 MMIO base
- magic/version/device_id/vendor_id
- RX/TX queue size
- negotiated features
- `DRIVER_OK`

### Task C2：修正 virtqueue 初始化和物理地址

**文件：**

- 修改：`kernel/net/drivers/virtio_net.cc`
- 修改：`kernel/net/drivers/virtio_net.hh`

**检查点：**

- RX queue 和 TX queue 的 desc/avail/used 地址必须按 virtio legacy 要求对齐。
- 写 `QUEUE_PFN` 时必须使用物理地址，不要把高地址内核虚拟地址右移后直接塞进去。
- buffer descriptor 的 `addr` 也必须是设备能 DMA 的物理地址。
- 关闭不支持的 offload：CSUM/GUEST_CSUM/TSO/MRG_RXBUF/INDIRECT/EVENT_IDX。
- 如果启用 `VIRTIO_NET_F_MAC`，从 config space `0x100` 起读取 MAC；读不到时使用本地管理地址，例如 `52:54:00:12:34:56`，并打印 fallback。

**验收：**

增加 `virtio_net_debug_status()` 输出：

- queue ready 状态
- rx_avail idx / rx_used idx
- tx_avail idx / tx_used idx
- MAC
- counters

### Task C3：实现 TX completion polling

**文件：**

- 修改：`kernel/net/drivers/virtio_net.cc`
- 修改：`kernel/net/drivers/virtio_net.hh`

**实现要求：**

新增内部函数：

```cpp
static void virtio_net_reclaim_tx_locked();
```

调用位置：

- `virtio_net_send()` 分配 TX descriptor 前调用一次，回收已完成 descriptor。
- `virtio_net_send()` notify 后可以短轮询几次，避免没有中断时 8 个 descriptor 很快耗尽。
- `virtio_net_intr()` 中也调用同一个回收函数。

**验收：**

用一个内核测试循环发送多于 `NUM_NET_DESC` 的短包，预期：

- 不会在第 9 个包后永久 `No free TX descriptors`
- `tx_completed` 会增长

### Task C4：实现 RX polling 和 adapter pump

**文件：**

- 修改：`kernel/net/drivers/virtio_net.cc`
- 修改：`kernel/net/drivers/virtio_net_adapter.cc`

**实现要求：**

1. `virtio_net_recv()` 保持非阻塞语义：
   - 没包返回负值
   - 有包复制 Ethernet frame，不包含 virtio-net header
   - 回收 RX descriptor
   - 立即补一个新的 RX descriptor

2. `virtio_net_adapter.cc` 实现循环：

```cpp
while (recv_thread_running)
{
    uint32 packet_len = sizeof(packet_buffer);
    int ret = net::virtio_net_recv(packet_buffer, &packet_len);
    if (ret == 0 && packet_len > 0)
    {
        ethernet_ii_recv(netif, packet_buffer, packet_len);
        continue;
    }
    os_sleep_ms(1);
}
```

3. 注意 `ethernet_add()` 的 receive thread starter 不能把初始化路径永久卡住。如果 `start_recv_thread_wrapper()` 会被同步调用，就不能直接在里面进入无限循环。需要创建真正的内核线程，或者改成只记录 netif，由网络栈统一启动 RX pump。

建议做法：

- 新增 `net::start_adapter_rx_pump(PST_NETIF netif)`。
- 内部使用一个内核线程辅助函数启动 RX pump。
- 如果暂时没有通用 kernel thread API，可以先抽出 `kernel/net/onpstack/port/os_adapter.cc` 里创建内核线程的逻辑，形成小型公共 helper，避免复制粘贴。

**验收：**

用 QEMU user net 运行时，日志应该看到：

- RX pump started
- 收到 ARP/IP/TCP 相关 frame 时 `rx_received` 增长
- 没有忙等刷屏

**文件：**

- 修改：`kernel/trap/riscv/plic.cc`
- 修改：`kernel/trap/riscv/trap.cc`

## 5. 阶段 D：把非 loopback socket 接到 ONPS

目标：用户态 Linux socket API 对非 loopback IPv4 TCP client 可用。

### Task D1：给 socket_file 增加 backend 状态

**文件：**

- 修改：`kernel/fs/vfs/file/socket_file.hh`
- 修改：`kernel/fs/vfs/file/socket_file.cc`

**设计：**

新增枚举：

```cpp
enum class SocketBackend
{
    NONE,
    LOOPBACK,
    ONPS
};
```

`socket_file` 增加字段：

```cpp
SocketBackend _backend;
int _last_error;
```

规则：

- AF_UNIX：继续走现有 UNIX socket 逻辑。
- AF_INET + loopback：`_backend = LOOPBACK`。
- AF_INET + 非 loopback：`_backend = ONPS`。
- backend 一旦进入 CONNECTED/BOUND/LISTENING 后不能随意切换。

### Task D2：封装 ONPS socket 创建和错误映射

**文件：**

- 修改：`kernel/fs/vfs/file/socket_file.cc`

**新增私有函数：**

```cpp
int ensure_onps_socket_locked();
int map_onps_error_to_errno(int onps_error) const;
bool should_use_loopback(const struct sockaddr_in &addr) const;
```

`ensure_onps_socket_locked()` 行为：

- 如果 `_onps_socket != INVALID_SOCKET`，直接返回 0。
- TCP socket 调 ONPS `socket(AF_INET, SOCK_STREAM, 0, &err)`。
- UDP socket 调 ONPS `socket(AF_INET, SOCK_DGRAM, 0, &err)`。
- 失败时保存 `_last_error`，返回 Linux 负 errno。

错误映射第一版：

- 地址族不支持 -> `-EAFNOSUPPORT`
- socket 类型不支持 -> `-EPROTONOSUPPORT`
- 连接超时 -> `-ETIMEDOUT`
- 连接被重置 -> `-ECONNRESET`
- 未连接 -> `-ENOTCONN`
- 无数据且非阻塞 -> `-EAGAIN`
- 其他 -> `-EIO`

### Task D3：实现非 loopback TCP connect/send/recv

**文件：**

- 修改：`kernel/fs/vfs/file/socket_file.cc`

**connect 规则：**

- 如果目标是 loopback，保持现有逻辑。
- 如果目标是非 loopback：
  - 确保网络栈已初始化；未初始化返回 `-ENETDOWN`。
  - `ensure_onps_socket_locked()`。
  - 调 ONPS `connect_ext(_onps_socket, &remote_addr.sin_addr, remote_port, timeout)`。
  - 成功后设置 `_remote_addr`、`_state = CONNECTED`、`_backend = ONPS`。

注意端口字节序必须用测试确认：

- Linux `sockaddr_in.sin_port` 是网络字节序。
- ONPS 的 `connect_ext()` 参数 `USHORT srv_port` 需要确认是主机序还是网络序。
- 计划执行时必须用抓包/日志确认 SYN 发往正确端口 80，而不是 20480。

**send 规则：**

- `_backend == LOOPBACK`：保持现有逻辑。
- `_backend == ONPS`：
  - TCP 用 ONPS `send()` 或 `send_nb()`。
  - 第一版阻塞 socket 可以使用带 ACK timeout 的 `send()`。
  - 非阻塞 socket 使用 `send_nb()`，缓冲区满返回 `-EAGAIN`。

**recv 规则：**

- `_backend == LOOPBACK`：保持现有逻辑。
- `_backend == ONPS`：
  - 根据 `_blocking` 设置 ONPS receive timeout。
  - blocking：可以短超时循环等待，但要能被 close/shutdown 唤醒。
  - nonblocking/`MSG_DONTWAIT`：无数据返回 `-EAGAIN`。

**验收：**

跑 `network_http_gateway_smoke()`。

预期：

- 如果宿主机 HTTP server 已启动，guest 能连到 `10.0.2.2:18080` 并收到 `HTTP/`。
- 日志中 TX/RX counters 增长。

### Task D4：实现非 loopback UDP sendto/recvfrom

**文件：**

- 修改：`kernel/fs/vfs/file/socket_file.cc`

**为什么要做：**

DNS 需要 UDP。即使第一阶段 HTTP by IP 不需要 DNS，后面 BusyBox `wget http://example.com/` 会走 libc resolver，resolver 会用 UDP 访问 DNS。

**sendto 规则：**

- loopback 目标继续走当前 UDP datagram queue。
- 非 loopback 目标：
  - `ensure_onps_socket_locked()`
  - ONPS `sendto(_onps_socket, ip_string 或 ip_ext, port, data, len)`
  - 返回实际发送字节数。

**recvfrom 规则：**

- loopback 保持现有逻辑。
- ONPS UDP 调 `recvfrom()`，把来源 IP/port 回填给用户。

**验收：**

- 写一个 DNS UDP smoke：向 `10.0.2.3:53` 发送一个 A 记录查询。
- 只要求收到合法 DNS response，不要求一开始就接入 libc `getaddrinfo()`。

### Task D5：loopback 并发检查点

**目标：**

在非 loopback backend 接入后，确认已经跑通的 loopback 后端仍能承受并发和吞吐，不被 ONPS 路径影响。

**推荐测例：**

- `sendmmsg01`
- `recvmmsg01`
- `send02`
- scoreboard `iperf.md` 里的 8 项
- scoreboard `netperf.md` 里的 6 项

**真实脚本来源：**

- `/mnt/sdcard-rv/musl/iperf_testcode.sh`
- `/mnt/sdcard-rv/musl/netperf_testcode.sh`

**注意：**

- `iperf_testcode.sh` 使用 `./iperf3`，不是名为 `iperf` 的二进制。
- `iperf` 和 `netperf` 脚本都以 `127.0.0.1` 为目标，主要验证 loopback 并发和吞吐，不验证真实外网。
- 如果 `netperf` timeout，要先看是后台 `netserver`、poll/select、阻塞唤醒、还是 socket 队列问题，不要直接归因到 virtio-net。

## 11. 风险和取舍

### 不先做 HTTPS

访问网页数据第一阶段只做 HTTP。HTTPS 需要 TLS 库、证书、随机数、时间和更多 libc/文件系统支持，不是“网络能收发数据”的最小闭环。

### 不先做 DHCP

QEMU user net 下静态 `10.0.2.15/24` 足够跑通第一阶段。DHCP 是后续易用性任务，不应挡住 HTTP by IP。

### 不把 ONPS 用于 loopback

当前 loopback 已经有真实 payload 证据，贸然替换会让已通过测试回退。短期保留 loopback backend，非 loopback 才走 ONPS。

### 不把 syscall_handler 重新写成 ONPS 大杂烩

`syscall_handler.cc` 只负责 Linux ABI 参数、fd、copy_in/copy_out、errno。真实 socket 状态和后端选择放在 `socket_file`，避免再出现两套状态机互相打架。

### 第一阶段允许 polling

中断是性能和干净架构需要，但不是“能访问到网页数据”的必要条件。先用 polling 跑通，再接 PLIC/PCI 中断，风险更低。
